### 2. `markOop` 定义为 `uintptr_t` 是一种合理的简化

```cpp
// my_jvm/src/utilities/globalDefinitions.hpp
typedef uintptr_t markOop;
```

**本地源码的真实情况**（`oopsHierarchy.hpp` 第 43 行）：
```cpp

```

OpenJDK 11 里 `markOop` 是 `markOopDesc*`，`markOopDesc` 继承自 `oopDesc`，通过 `value()` 方法把 `this` 指针转成 `uintptr_t` 来做位操作：
```cpp
// markOop.hpp
class markOopDesc: public oopDesc {
  uintptr_t value() const { return (uintptr_t) this; }
  // ...
};
```

你用 `typedef uintptr_t markOop` 是一种**合理的简化设计**，避免了 `markOopDesc` 继承 `oopDesc` 的复杂性。但需要知道这和 OpenJDK 11 的真实实现不同——OpenJDK 11 里 `markOop` 是指针，不是整数值。

```cpp
inline jint atomic_cas(jint* addr, jint exchange_val, jint compare_val) {
```

## ❌ 仍然存在的问题（严格基于本地源码）

### 问题 1：`unlocked_value` 和 `locked_value` 的值与本地源码相反（语义错误）

**你的代码**（`markOop.hpp`）：
```cpp
enum {
    unlocked_value  = 0,   // ← 你认为 0 是未锁定
    locked_value    = 1,   // ← 你认为 1 是锁定
    ...
};
```

**本地源码**（`markOop.hpp`）：
```cpp
enum { locked_value             = 0,   // ← 0 是轻量级锁定（ptr | 00）
       unlocked_value           = 1,   // ← 1 是未锁定（header | 01）
       monitor_value            = 2,   // ← 2 是重量级锁（ptr | 10）
       marked_value             = 3,   // ← 3 是 GC 标记（ptr | 11）
       biased_lock_pattern      = 5    // ← 5 是偏向锁（...| 101）
};
```

本地源码的注释解释了原因：
```
[ptr             | 00]  locked             ptr points to real header on stack
[header      | 0 | 01]  unlocked           regular object header
[ptr             | 10]  monitor            inflated lock
[ptr             | 11]  marked             used by markSweep
```

低2位 `00` 表示**轻量级锁定**（因为栈上的 `BasicLock` 指针低位对齐，低2位自然为0）。低2位 `01` 才是**未锁定**的普通对象头。

**修复方法**：
```cpp
enum {
    locked_value        = 0,   // 轻量级锁定：低2位 00
    unlocked_value      = 1,   // 未锁定：低2位 01
    monitor_value       = 2,   // 重量级锁：低2位 10
    marked_value        = 3,   // GC 标记：低2位 11
    biased_lock_pattern = 5    // 偏向锁：低3位 101
};
```

### 问题 2：`markWord_hash` 的提取逻辑错误

**你的代码**（`markOop.hpp`）：
```cpp
inline uintptr_t markWord_hash(markOop m) {
    return (m >> 5) << 5;   // ← 这只是清除了低5位，不是提取哈希码
}
```

`(m >> 5) << 5` 的结果是把低 5 位清零，返回的还是一个 mark word 形状的值，不是哈希码本身。

**本地源码**（`markOop.hpp`）：
```cpp
// hash_shift = cms_shift + cms_bits = (age_shift + age_bits) + 1
//            = (lock_bits + biased_lock_bits + age_bits) + 1
//            = (2 + 1 + 4) + 1 = 8
// hash_bits  = max_hash_bits > 31 ? 31 : max_hash_bits = 31（64位下）
// hash_mask  = right_n_bits(31) = 0x7FFFFFFF

intptr_t hash() const {
    return mask_bits(value() >> hash_shift, hash_mask);
    // = (value() >> 8) & 0x7FFFFFFF
}
```

**修复方法**：
```cpp
inline uintptr_t markWord_hash(markOop m) {
    return (m >> 8) & 0x7FFFFFFF;  // hash_shift=8, hash_bits=31
}
```

### 问题 3：`markWord_with_hash` 的 shift 值错误

**你的代码**（`markOop.hpp`）：
```cpp
inline markOop markWord_with_hash(markOop m, uintptr_t hash) {
    return (m & ~((uintptr_t)0x3F)) | (hash << 5);  // ← shift=5 是错的
}
```

**本地源码**（`markOop.hpp`）：
```cpp
markOop copy_set_hash(intptr_t hash) const {
    intptr_t tmp = value() & (~hash_mask_in_place);  // 清除 hash 位
    tmp |= ((hash & hash_mask) << hash_shift);        // hash_shift = 8
    return (markOop)tmp;
}
```

`hash_shift = 8`，不是 5。你用 `hash << 5` 会把哈希码写到错误的位置，与 `age` 字段重叠。

**修复方法**：
```cpp
inline markOop markWord_with_hash(markOop m, uintptr_t hash) {
    uintptr_t hash_mask_in_place = ((uintptr_t)0x7FFFFFFF) << 8;
    return (m & ~hash_mask_in_place) | ((hash & 0x7FFFFFFF) << 8);
}
```

### 问题 4：`markWord_biased` 的参数设计与本地源码不符

**你的代码**（`markOop.hpp`）：
```cpp
inline markOop markWord_biased(char* obj, int epoch, Klass* klass) {
    uintptr_t ptr = (uintptr_t)klass;   // ← 用的是 Klass*，错误！
    ptr = (ptr << 10) | ((uintptr_t)epoch << 8) | ((uintptr_t)4) | bias_pattern;
    return (markOop)ptr;
}
```

**本地源码**（`markOop.hpp`）：
```cpp
static markOop encode(JavaThread* thread, uint age, int bias_epoch) {
    intptr_t tmp = (intptr_t) thread;   // ← 用的是 JavaThread*，不是 Klass*！
    assert(UseBiasedLocking && ((tmp & (epoch_mask_in_place | age_mask_in_place | biased_lock_mask_in_place)) == 0), ...);
    return (markOop) (tmp | (bias_epoch << epoch_shift) | (age << age_shift) | biased_lock_pattern);
}
```

偏向锁的 mark word 格式（64位）：
```
[JavaThread*:54 | epoch:2 | unused:1 | age:4 | biased_lock:1 | lock:2]
```

偏向的是**线程**（`JavaThread*`），不是 `Klass*`。`obj` 参数完全没用到，`age` 写死成了 4，这两个问题都需要修复。

### 问题 5：`globalDefinitions.hpp` 里 `u1/u2` 类型定义错误

**你的代码**（`globalDefinitions.hpp`）：
```cpp
typedef uint32_t  u4;   // ✅ 正确
typedef uint64_t  u8;   // ✅ 正确
typedef int32_t   u1;   // ❌ 错误！u1 应该是无符号 1 字节
typedef int16_t   u2;   // ❌ 错误！u2 应该是无符号 2 字节
```

**本地源码**（`globalDefinitions.hpp`）：
```cpp
// Unsigned one, two, four and eight byte quantities used for describing
// the .class file format. See JVM book chapter 4.
typedef jubyte  u1;    // jubyte  = uint8_t
typedef jushort u2;    // jushort = uint16_t
typedef juint   u4;    // juint   = uint32_t
typedef julong  u8;    // julong  = uint64_t
```

`u` 代表 unsigned，`u1/u2` 是 class 文件解析用的无符号类型，用有符号的 `int32_t`/`int16_t` 是错的。

**修复方法**：
```cpp
typedef uint8_t   u1;
typedef uint16_t  u2;
typedef uint32_t  u4;
typedef uint64_t  u8;
```

### 问题 7：`typedef jint jint` 自我重定义（编译报错）

**你的代码**（`globalDefinitions.hpp`）：
```cpp
typedef int32_t  jint;    // 第一次定义：jint = int32_t ✅
// ...
typedef jint     jint;    // 第二次定义：jint = jint ← ❌ 自我重定义，编译报错！
typedef jlong    jlong;   // 同上 ❌
```

C++ 不允许把一个类型 `typedef` 成自身（即使类型相同也不行）。这会直接导致编译失败。

**修复方法**：删掉第二次的 `typedef jint jint` 和 `typedef jlong jlong`。
