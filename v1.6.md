## ❌ 仍然存在的问题（严格基于本地源码）

### 问题 1：`ConstMethod` 缺少 `_constants` 字段（最重要的字段）

**你的代码**（`constMethod.hpp`）：
```cpp
class ConstMethod : public Metadata {
    u2 _name_index;
    u2 _signature_index;
    u1* _code;          // ← 用裸指针存字节码
    u4 _code_size;
    u2 _max_locals;
    u2 _max_stack;
    // ...
};
```

**本地源码**（`constMethod.hpp` 第 200 行）：
```cpp
ConstantPool*  _constants;    // ← 第一个字段！常量池指针
Array<u1>*     _stackmap_data;
// ...
u2             _code_size;
u2             _name_index;
u2             _signature_index;
```

你的 `ConstMethod` 缺少了最重要的字段 `_constants`（常量池指针）。`_name_index`、`_signature_index` 都是常量池里的索引，没有 `_constants` 就无法解析方法名和签名。

另外，本地源码里字节码不是用 `u1*` 裸指针存的，而是**紧跟在 `ConstMethod` 结构体末尾内联存放**的（见本地源码注释：`The actual bytecodes are inlined after the end of the ConstMethod struct`）。这是一个重要的内存布局设计。

**修复方法**：
```cpp
class ConstMethod : public Metadata {
    ConstantPool*  _constants;      // ← 补上常量池指针（第一个字段）
    Array<u1>*     _stackmap_data;  // ← 可选
    int            _constMethod_size;
    u2             _flags;
    u2             _code_size;
    u2             _name_index;
    u2             _signature_index;
    u2             _max_stack;
    u2             _max_locals;
    u2             _size_of_parameters;
    // 字节码内联在结构体末尾，不用裸指针
};
```

### 问题 2：`ConstMethod` 用 `u1* _code` 裸指针存字节码，与本地源码设计不符

**你的代码**：
```cpp
u1* _code;    // 字节码数组（裸指针）
u4 _code_size;
```

**本地源码的真实设计**（`constMethod.hpp` 注释第 40-45 行）：
```
// The actual bytecodes are inlined after the end of the ConstMethod struct.
//
// The line number table is compressed and inlined following the byte codes.
```

本地源码里字节码是**内联在 `ConstMethod` 结构体末尾**的，通过 `code_base()` 方法获取：
```cpp
// constMethod.hpp 第 380 行
address code_base() const {
    return (address) (this + 1);   // ← this+1 就是字节码起始地址！
}
```

这个设计的好处是：`ConstMethod` 和字节码在内存中是连续的，一次 Metaspace 分配就搞定，减少内存碎片，提升缓存命中率。

你用 `u1* _code` 裸指针是一种简化，但要知道这和本地源码的内存布局完全不同。

### 问题 3：`Array<T>` 缺少 Metaspace 分配机制

**你的代码**（`array.hpp`）：
```cpp
template <typename T>
class Array : public MetaspaceObj {
    int _length;
    T   _data[1];
    // 没有 operator new！
};
```

**本地源码**（`array.hpp` 第 55-60 行）：
```cpp
void* operator new(size_t size, ClassLoaderData* loader_data, int length, TRAPS) throw() {
    size_t word_size = Array::size(length);
    return (void*) Metaspace::allocate(loader_data, word_size,
                                       MetaspaceObj::array_type(sizeof(T)), THREAD);
}
```

本地源码的 `Array<T>` 重载了 `operator new`，强制从 Metaspace 分配内存，而不是从堆上分配。你的版本没有这个机制，直接 `new Array<T>` 会从堆上分配，这和 JVM 的内存管理模型不符。

这是一个**设计层面的差异**，不是简单的字段缺失。如果要推进类加载，这个问题迟早要面对。
